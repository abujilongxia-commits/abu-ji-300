# 阿布吉300任務網站 — 需求分析文檔

**文件版本：** 1.0  
**建立日期：** 2026年4月2日  
**負責角色：** 宸瑋（客戶關係守護者）

---

## 1. 目標受眾分析

### 1.1 主要用戶群體

| 用戶類型 | 描述 | 主要需求 |
|---------|------|---------|
| **任務管理者** | 需要系統化管理大量任務的專業人士 | 高效的任務分類、搜尋、追蹤 |
| **專案經理** | 跨團隊協作的領導者 | 進度視覺化、權責分配、截止日期管理 |
| **自由工作者** | 個人時間管理者 | 輕量化、易上手、跨裝置同步 |
| **學生/研究者** | 學術任務規劃者 | 里程碑設定、檔案關聯、筆記整合 |

### 1.2 用戶特徵

- **年齡區間：** 18-55歲
- **技術熟悉度：** 中等（熟悉網頁應用操作）
- **使用情境：** 桌面為主（70%），行動裝置為輔（30%）
- **語言需求：** 繁體中文為主，未來支援多語言

### 1.3 使用者痛點

1. 現有任務管理工具過於複雜學習曲線高
2. 跨裝置同步不及時導致資訊落差
3. 缺乏足夠的視覺化進度呈現
4. 搜尋與篩選功能不夠直覺

## 3. 使用者故事 (User Stories)

### 3.1 核心流程

| # | 使用者故事 | 情境 |
|---|-----------|------|
| US-01 | 作為**任務管理者**，我希望**快速建立任務**，以便**節省時間而不犧牲結構** | 點擊「+」按鈕 → 輸入標題 → Enter確認 |
| US-02 | 作為**專案經理**，我希望**一眼看到所有任務的進度**，以便**快速識別瓶頸** | 開啟儀表板 → 查看完成率/逾期率圖表 |
| US-03 | 作為**自由工作者**，我希望**跨裝置同步我的任務**，以便**隨時隨地掌握進度** | 手機建立 → 桌面立馬可見 |
| US-04 | 作為**學生**，我希望**用標籤區分科目**，以便**快速篩選當週作業** | 點選「數學」標籤 → 只顯示數學相關任務 |
| US-05 | 作為**使用者**，我希望**復原誤刪的任務**，以便**避免資料損失恐慌** | 刪除 → 出現在「回收筒」→ 一鍵復原 |

### 3.2 邊界情境

| # | 使用者故事 | 情境 |
|---|-----------|------|
| US-06 | 作為**使用者**，我希望**搜尋結果即時呈現**，以便**不用等待載入** | 輸入關鍵字 → 100ms內顯示結果 |
| US-07 | 作為**使用者**，我希望**截止日逾期時有明顯提示**，以便**不會錯過重要期限** | 逾期任務顯示紅色標記 + 儀表板警告 |

## 4. 非功能需求

### 4.1 效能需求

| 指標 | 目標值 | 說明 |
|-----|-------|------|
| 頁面首次載入時間 | ≤ 2秒 | Lighthouse Performance Score ≥ 90 |
| 搜尋回應時間 | ≤ 200ms | 10,000筆資料以內 |
| 操作回饋延遲 | ≤ 100ms | 使用者點擊到視覺反饋 |
| 同時在線人數 | ≥ 100人 | 預估初期規模 |

### 4.2 安全需求

| 項目 | 需求 |
|-----|------|
| 身份驗證 | JWT Token，Access Token 15分鐘過期，Refresh Token 7天 |
| 密碼儲存 | bcrypt hash，cost factor ≥ 12 |
| 資料傳輸 | 全站 HTTPS (TLS 1.3) |
| CORS 政策 | 嚴格限定允許的 origin |
| 輸入驗證 | 所有輸入做 XSS/SQL Injection 防護 |
| Rate Limiting | API 端點限流，防止暴力破解 |

### 4.3 易用性需求

| 項目 | 需求 |
|-----|------|
| 響應式設計 | Mobile-first，支援 320px ~ 2560px |
| 鍵盤快捷鍵 | 主要操作皆可鍵盤完成（如：N 新增任務，/ 搜尋） |
| 無障礙支援 | WCAG 2.1 AA 等級 |
| 多語言框架 | i18n 預留，未來可擴展 |
| 離線支援 | Service Worker 快取，恢復連線後同步 |

### 4.4 可維護性需求

| 項目 | 需求 |
|-----|------|
| 程式碼覆蓋率 | 單元測試 ≥ 80% |
| API 文件 | OpenAPI 3.0 規格 |
| 日誌系統 | 結構化日誌 (JSON format) |
| 監控告警 | 錯誤率 > 1% 自動告警 |

### 4.5 擴展性需求

| 項目 | 需求 |
|-----|------|
| 水平擴展 | 無狀態設計，可新增 API Server 擴容 |
| 資料庫分片 | 使用 UUID 作為主鍵，未來可依用戶 ID 分片 |
| 快取層 | Redis 快取熱門資料，降低資料庫負載 |
